# 節の下書き生成

骨組みの状態（`## 見出し` + `<!-- メモ -->`）の節を、CLAUDE.md のルールに沿った下書きに変換する。

## Step 1: 入力の確認

以下を特定する。

- **対象の記事ファイル**（明示されなければ `articles/` 内から選んでもらう）
- **対象の節**（見出し名 or 行番号）
- **メモ**（節内の HTML コメントから読む。なければユーザーから聞き取る）
- **想定読者**（記事全体のテーマと frontmatter の `topics` から推測。曖昧なら確認）

## Step 2: 文脈を読む

- 記事ファイルを読み、対象節の **前後の節** を把握する（流れを壊さないため）
- frontmatter の `title` を確認する（タイトルとの整合性を保つ）
- CLAUDE.md の「記事執筆のルール」を再確認する

## Step 3: 下書きを生成

以下を守って下書きを書く。

### 守るべきルール

CLAUDE.md の「記事執筆のルール」（文体・表記・Markdown 記法）にすべて従う。節内で小見出しが必要なら `###` を使う（`##` は節の見出しが担う）。

### 内容の方針

- メモに書かれていない技術詳細を **勝手に作らない**（事実をでっち上げない）
  - 必要な情報がメモにない場合は、ユーザーに質問するか「ここは追記してください」と明示する
- 節のサイズは内容に応じて適切に（無理に膨らませない）
- 冗長表現を避ける（「〜することができる」→「〜できる」）
- 文末の同じ語尾を連続させない
- コードや設定値は、リポジトリ内の実ファイル（`Dockerfile`、`devcontainer.json` など）と整合させる

## Step 4: 下書きの提示

下書きをチャット上に提示する。

```
## [節の見出し]

[下書き本文]
```

加えて以下を伝える。

- 推測で書いた箇所があれば明示する（「ここは推測です。実際の値に置き換えてください」など）
- メモに無くて補ったほうが良い情報があれば指摘する

## Step 5: ファイルへの反映

ユーザーが「OK」「書き込んで」と承認したら、対象節の `<!-- メモ -->` を下書きに置き換える形でファイルを更新する。

- HTML コメントが無い節の場合は、見出しの下に下書きを挿入する
- 既存の本文がある節を上書きする際は、必ず確認を取る

## 重要な制約

- メモや既存の事実に書かれていない技術内容を **創作しない**
- ユーザー承認なしに既存本文を上書きしない
- 一度に複数の節を書こうとしない（1 節ずつ確認しながら進める）
- 文末を機械的に整えるだけの修正には使わない（それは編集者視点の `zenn-content-reviewer` か直接の編集依頼で）
- 推測した内容は明示的にラベルする（「推測：」「要確認：」など）
